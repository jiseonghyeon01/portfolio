About Me

TodoList 서비스

개요

개인 일정 관리를 위해 개발한 웹 기반 투두리스트 애플리케이션입니다.

  • 프로젝트 명:TodoList
  • 개발 인원: 5명 (팀 프로젝트)
  • 개발 기간: 2025.08.13 ~ 2025.08.30
Spring Framework java Mysql Thymeleaf AWS EC2 JDBCTemplate

[나의 주요 역할]

저는 프로젝트의 핵심인 사용자 인증 (로그인) 및 데이터 단일 조회 기능의 구현 및 테스트를 담당했습니다.

  • 로그인 기능 구현: 로그인 View(Thymeleaf), Controller, UserService(`findUserById`), UserDAO(`selectUserById`)에 걸친 전체 흐름을 구현했습니다.
  • 세션/권한 관리: 로그인 성공 시 세션 생성 로직을 처리하여 인가(Authorization)의 기초를 마련하고, 로그인 여부에 따른 접근 권한 제어를 담당했습니다.
  • 단일 Todo 조회: ReadTodo(상세 조회) 기능을 구현하여 본인 Todo만 열람 가능하도록 Service 계층에서 접근 권한 로직을 처리했습니다.
  • 테스트 주도 개발: 구현한 모든 기능에 대해 단위 테스트 및 통합 테스트를 선행/병행하며 코드 신뢰도를 확보했습니다.

[트러블 슈팅]

로그인 및 조회 기능을 구현하는 과정에서, UserService가 UserDAO에 직접 의존하는 구조로 인해 단위 테스트가 어려워지는 문제가 발생했습니다.

  • 문제 발생: UserDAO를 Mocking 없이 테스트하기 때문에 DB 설정이 불안정하거나 변경될 때마다 UserService의 테스트가 실패하는 결합도 문제가 발생했습니다.
  • 해결 방안 (SOLID 적용): 의존성 역전 원칙(DIP)에 따라 UserService가 구체적인 UserDAO 클래스가 아닌, UserDAO 인터페이스에 의존하도록 구조를 변경했습니다.
  • 결과: UserService의 로직(비즈니스 로직)과 UserDAO의 DB 접근 방식이 분리되어, Mockito를 사용하여 DAO의 동작을 모킹(Mocking)할 수 있게 되었고, DB 없이 UserService의 순수 단위 테스트를 성공적으로 확립했습니다.

[성과 및 배운 점]

본 프로젝트를 통해 견고한 백엔드 구현 역량과 신입 개발자로서 갖춰야 할 설계 원칙 및 테스트 방법론을 체계적으로 습득했습니다.

  • SOLID 원칙 및 테스트 역량 강화: 의존성 역전 원칙을 적용한 트러블 슈팅 경험을 통해, 이론적 설계 원칙이 실제 코드 품질과 테스트 용이성에 미치는 영향을 깊이 있게 이해했습니다. 이를 통해 UserService의 순수 단위 테스트 환경을 확립하며, 재사용 가능하고 유지보수가 쉬운 코드를 작성하는 방법을 배웠습니다.
  • 인증/인가 흐름 및 3-Tier 이해: 로그인 성공 시 세션 생성 및 본인 Todo만 열람 가능하도록 권한을 제어하는 로직을 직접 구현하며, Controller-Service-DAO로 이어지는 3-Tier 아키텍처 내에서 사용자 인증과 인가 흐름을 명확히 체화했습니다.
  • 결론: 단순 기능 구현을 넘어, 테스트 커버리지와 코드 품질을 최우선으로 생각하는 개발자로 성장할 수 있는 기반을 다진 프로젝트입니다.

스터디룸 예약 시스템

개요

사용자들이 스터디룸을 검색하여 카카오맵어플로 스터디룸을 찾아 편리하게 예약하고,리뷰를 쓸수 있고,사용자끼리 게시판도 이용할수도 있습니다.

  • 프로젝트 명: Studyroom-reserve-system
  • 개발 인원: 8명 (팀 프로젝트)
  • 개발 기간: 2025.09.26 ~ 2023.10.30
Spring Framework java Mysql Thymeleaf AWS EC2 JPA

[나의 주요 역활]

저는 사용자 인증 흐름, 커뮤니티(리뷰) 기능의 백엔드 로직, 그리고 예약 데이터 조회 파트를 담당하며 시스템의 안정성과 기능 구현의 완성도를 높이는 데 집중했습니다.

특히, Spring Security와 JPA를 통합하여 회원가입/로그인/로그아웃 기능 및 세션 관리를 구현했습니다. 인증 성공 시 세션을 생성하고, **단일 예약 및 리뷰 조회 시 해당 사용자의 접근 권한을 Service 계층에서 검증하는 로직을 담당하여 인가(Authorization)의 기초를 다졌습니다.

또한, 리뷰 작성/수정/삭제/신고 기능은 UML 다이어그램에 명시된 `ReviewController` - `ReviewService` - `ReviewRepository`의 명확한 요청 흐름에 따라 구현되었으며, 특히 신고 및 삭제 시 권한 검증 로직을 포함했습니다.

[트러블 슈팅]

JPA 환경에서 Spring Security의 사용자 인증을 구현하는 과정에서 트러블 슈팅이 필요했습니다. `UserDetailsService`가 데이터베이스에서 조회한 `UserEntity`와 Spring Security가 요구하는 `UserDetails` 객체 간의 변환 및 매핑에서 문제가 발생했습니다.

문제의 본질은 JPA 엔티티를 직접 보안 컨텍스트에 사용할 경우 발생할 수 있는 데이터 불일치 및 영속성 컨텍스트 침범 문제였습니다.

해결 방법으로, `UserDetailsService`를 커스터마이징하고, `UserEntity`를 조회한 후에는 `UserDetails` 인터페이스를 구현하는 별도의 DTO 객체로 변환하여 Spring Security 컨텍스트에 저장하도록 구조를 변경했습니다. 이 해결책을 통해 JPA의 설계 원칙을 준수**하면서도 보안 프레임워크의 요구사항을 만족시키는 안정적인 인증 시스템을 구축할 수 있었습니다.

[기술적 결정 및 성과]

이 프로젝트를 통해 객체지향 설계와 최신 프레임워크 활용 역량을 효과적으로 키웠습니다.

이전 프로젝트의 JDBCTemplate 방식 대신 JPA를 도입하여 엔티티 설계와 객체-관계 매핑(ORM)에 대한 깊은 이해를 얻었습니다. 이로써 SQL 작성 대신 엔티티 관계 중심으로 데이터 로직을 처리하는 현대적인 개발 생산성을 체감했습니다.

또한, UML 다이어그램을 기반으로 개발을 진행하여 Controller-Service-Repository의 계층 구조 및 역할 분리를 철저히 지켰습니다. 이는 코드를 읽기 쉽고, 유지보수가 용이하며, 향후 기능 확장이 용이한 구조적 사고방식을 체화하는 데 결정적인 도움이 되었습니다.
